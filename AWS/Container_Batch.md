# AWSコンテナ・バッチ処理学習記録

## 学習概要
- コンテナサービスの役割分担 (ECS, EKS, Fargate)
- バッチ処理の概念とAWS Batch

## コンテナサービスの構成要素
コンテナを動かすには「司令塔」と「実行場所」の2つが必要。

### 1. 司令塔（オーケストレーター）
- **ECS (Elastic Container Service)**:
    - AWS純正の司令塔。
    - 他のAWSサービス（ALB, IAM）との連携が強力で簡単。**AWSのスタンダード**。
- **EKS (Elastic Kubernetes Service)**:
    - Kubernetes (k8s) のAWSマネージドサービス。
    - OSS標準で高機能だが、設定が複雑で学習コストが高い。

### 2. 実行場所（データプレーン）
- **Fargate**:
    - **サーバーレス**。サーバー（EC2）の管理が一切不要。
    - 「ホテル」のイメージ。部屋の掃除はAWSがやる。
    - 今の主流は **ECS + Fargate** の組み合わせ。
- **EC2**:
    - 自分でサーバーを管理する。
    - コスト調整やGPU利用など、細かい制御が必要な場合に使う。

## バッチ処理とAWS Batch
- **バッチ処理とは**:
    - 「まとめて、後で、一気にやる処理」。
    - 例：給与計算、日次集計、動画エンコード。
    - 対義語は「リアルタイム処理（Webアプリなど）」。
- **AWS Batch**:
    - バッチ処理を実行するための環境を全自動で提供するサービス。
    - **機能**:
        - 必要な時だけEC2を大量に起動し、終わったら自動で停止する。
        - 安価なスポットインスタンスを自動活用してコストを削減する。
        - 失敗時のリトライ機能など。

## 現時点での理解レベル
- コンテナは「司令塔（ECS/EKS）」と「場所（Fargate/EC2）」の組み合わせで考える。
- Fargateは「サーバー管理したくない」時の正解。
- バッチ処理は「閉店後の皿洗い」で、AWS Batchはそれを全自動でやってくれる「ロボット軍団」。
